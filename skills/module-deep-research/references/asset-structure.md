# 模块认知资产结构

模块认知资产位于：

```text
.sdd/modules/<module>/
  <模块名>模块设计说明书.md   # 总纲 + 功能概览索引
  业务功能/                    # 功能点的家，下分两个子目录
    业务流程/                  # 端到端流程类功能点，一个文件一个流程
    非流程功能点/              # 非流程功能点（横切能力、特殊场景处理），一个文件一个
  数据模型与ER图.md           # 数据库表、字段语义、表间关系（含 ER 图）
  约束与风险.md               # 约束规则 + 风险（无运作过程的纯规则）
  研究过程/                    # 过程文档隔离存放，保持根目录清爽
    研究队列.md               # 自循环驱动队列
    待解问题.md               # 未解问题（需外部输入）
```

## 设计取舍

资产按"认知产出"组织，不维护运行状态机和过程日志：

- **认知产出与过程文档分离**：根目录是稳定的模块认知（说明书、业务功能、数据模型、约束与风险），过程性的队列和待解问题隔离到 `研究过程/`。
- 证据锚点直接内嵌在认知文件的具体结论里，不单列证据索引——证据脱离结论无意义。
- 覆盖信息体现在 `研究过程/研究队列.md` 中每个研究任务的覆盖区域标注上，不单列覆盖地图。

> **命名澄清**：`<模块名>模块设计说明书.md` 描述的是这个模块"是什么、怎么运转"的**客观认知**（基于代码现状），不是任何针对具体需求或变更的**设计方案**。若同一目录下存在其他带"详细设计""设计方案"字样的文档，那是面向不同目的的独立文档，不要混淆。

## 业务功能的组织原则

`业务功能/` 下**一个功能点一个文件**，按"是不是端到端流程"分两个子目录：

- `业务功能/业务流程/`：**端到端流程类**功能点。有完整调用链的主路径（如"下发流程""资产同步"），一个文件一个流程。
- `业务功能/非流程功能点/`：**非流程类**功能点。横切能力（如"审计日志""鉴权"）、特殊场景处理（如"云核场景下需要校验插件包"），一个文件一个。

**判准**：这个功能点是不是端到端流程？是 → 进 `业务流程/`；不是 → 进 `非流程功能点/`。

两个子目录**启动时都建立**（即使第一次没探索到非流程功能点，`非流程功能点/` 这个结构也要在，避免后续补充时连结构都丢了）。

> 判断"一个点该进 `业务功能/` 还是 `约束与风险.md`"：它**有没有运作过程要讲**？有"怎么做"的过程（触发/处理/异常）进 `业务功能/`；只是一条不能破坏的纯规则、没有运作过程，进 `约束与风险.md`。两者可交叉引用，但归属按此一刀切。

## 排版与强调约定

- 文档内部章节统一使用**带序号的标题**（`## 1.` / `## 2.` / `### 2.1`），便于跨文档引用定位。
- 允许且推荐使用 markdown 字体颜色和突出显示作为强调：
  - 关键结论、风险、阻塞项用 `**加粗**` 或 `<mark>高亮</mark>` 突出。
  - 强约束、不可破坏的规则可用 `<span style="color: red">红色</span>` 标注。
  - 强调用于真正需要醒目的少量关键信息，不要整段染色。
- 画图推荐使用 **mermaid**：流程图、调用链、状态机、时序图、ER 图、依赖图等优先用 mermaid 表达，而不是长段文字。
  - **mermaid 图中所有节点和边的文本必须用双引号包裹转义**，避免中文/特殊字符破坏渲染。例如 `A["退款入口"] --> B["写入审计"]`。

## 写入原则

- 增量更新已有资产，不整篇重写到丢失既有判断。
- 只沉淀稳定认知、证据锚点、低置信推断、风险和未解问题。
- 不把临时推理过程写成长期事实。
- 不生成大而空的表格；无内容时保持简短说明。
- 缺失文件初始化为可继续追加的骨架，不等待完整信息齐备。

## 结论状态

所有重要结论必须标记状态：

- `FACT`：有代码、测试、文档或配置直接支撑。
- `INFERRED`：由多个信号合理推出，但没有直接明示。
- `CONFIRMED`：用户、团队文档或权威记录确认。
- `BLOCKED`：现有材料无法确认，且影响理解。

重要结论应包含：

- 证据锚点：文件路径、函数/类名、测试名或配置项。能定位行号时优先记录到 `文件:行号`。
- 影响范围：该结论适用的路径、场景、接口、数据或状态。

## 文件职责

- `<模块名>模块设计说明书.md`：模块总纲。涵盖职责与边界、依赖关系（外部+内部）、代码结构、设计模式与关键抽象、核心入口、数据模型概览、关键日志点、功能概览索引。**顶部固定记录「最后同步提交」**（运行结束时写入 `git rev-parse HEAD`），作为整个认知库的基线锚点；这是只读元信息，不触发自动刷新。
- `业务功能/`：功能点的家，下分两个子目录：
  - `业务功能/业务流程/`：端到端流程类功能点，每个文件承载一个流程的完整认知（概述/触发/处理流程/约束/异常/证据）。
  - `业务功能/非流程功能点/`：非流程类功能点（横切能力、特殊场景处理），每个文件承载一个功能点的完整认知。
  文件内开头说明这个功能点是什么，归属哪个子目录由"是不是端到端流程"判断。
- `数据模型与ER图.md`：模块涉及的数据库表清单、字段语义和用途、表与表之间的关系（主外键、关联、级联）。关系用 mermaid ER 图表达。
- `约束与风险.md`：纯约束规则（无运作过程的，如"退款金额≤原订单"）+ 风险（高复杂度、强耦合、冷路径、隐式约定等不宜轻动区域）。
- `研究过程/研究队列.md`：自循环驱动队列，记录候选研究任务、价值、状态、覆盖区域、证据入口和下一步动作。
- `研究过程/待解问题.md`：仍不理解、无法确认或需要外部裁决的问题。

## 初始骨架

### <模块名>模块设计说明书.md

```text
# <模块名>模块设计说明书

## 1. 当前状态
- 状态：待研究
- 最后更新：
- 最后同步提交：<git rev-parse HEAD 的结果，无法获取时填"无法获取">

## 2. 模块职责与边界
（模块是什么、解决什么问题、边界在哪、不做什么）

## 3. 依赖关系
### 3.1 上游依赖（依赖的外部模块/库/中间件/服务）
### 3.2 下游暴露（被哪些外部模块/服务消费）
### 3.3 内部依赖（模块内组件间的依赖结构）
### 3.4 依赖图（mermaid，外部+内部都画，节点文本双引号转义）

## 4. 代码结构
（分层组织/核心包职责/关键命名约定；只记架构关键点，不罗列每个文件）

## 5. 设计模式与关键抽象
（实际采用的设计模式+代码证据+采用原因；核心接口/基类/领域模型；值得注意的架构决策）

## 6. 核心入口与关键依赖

## 7. 数据模型概览
（指向《数据模型与ER图.md》）

## 8. 关键日志点
（关键业务节点的日志位置 logger/路径；关键业务报错 ERROR/WARN 对应的问题。只记"出问题时第一时间看哪里"）

## 9. 功能概览
| 功能点 | 类型 | 简述 | 文档 |
|--------|------|------|------|
| <名称> | 端到端流程 | 一句话 | 业务功能/业务流程/<文件>.md |
| <名称> | 横切能力/特殊场景 | 一句话 | 业务功能/非流程功能点/<文件>.md |

## 10. 低置信 / 待验证（INFERRED）

## 11. 阻塞（BLOCKED）
```

### 业务功能/ 功能点文件（业务流程/ 与 非流程功能点/ 通用骨架）

```text
# <功能点名>

## 1. 概述
（这个功能是什么；属于业务能力 / 支撑能力 / 特殊场景）

## 2. 触发条件 / 入口

## 3. 处理流程
（怎么做，含分支/状态变化/副作用）

## 4. 本功能的约束
（若有，如"审计日志不可删改"；纯规则也可在《约束与风险.md》记一条并互相指向）

## 5. 失败 / 异常路径

## 6. 证据锚点
```

### 数据模型与ER图.md

```text
# 数据模型与ER图

## 1. 表清单
（模块涉及哪些表、每张表的职责）

## 2. 字段语义
（每张表的关键字段、类型、用途；状态字段说明状态机流转）

## 3. 表间关系（ER 图）
（用 mermaid erDiagram 表达表与表之间的关系）

## 4. 证据锚点
```

> 若本模块不涉及数据库表（纯计算、纯无状态服务、纯前端等），`数据模型与ER图.md` 顶部写明"本模块不涉及数据库表"并简述原因，不强行编造。

### 约束与风险.md

```text
# 约束与风险

## 1. 约束规则
（无运作过程的纯规则：什么不能破坏、违反后果、证据；可用颜色/高亮强调强约束）

## 2. 风险
（高复杂度、强耦合、冷路径、隐式约定等不宜轻动区域：风险点+触发条件+原因）

## 3. 证据锚点
```

### 研究过程/研究队列.md

```text
# 研究队列

## 1. 候选研究任务

| ID | 研究任务 | 价值 | 状态 | 覆盖区域 | 触发原因 | 证据入口 | 下一步 |
|----|------|------|------|----------|----------|----------|--------|

## 2. 字段说明

- ID：稳定编号。
- 研究任务：具体到可取证的路径、场景、组件或问题。
- 价值：高 / 中 / 低。
- 状态：待研究 / 研究中 / 已完成 / 阻塞 / 延后。
- 覆盖区域：该研究任务覆盖的模块区域（兼作覆盖地图）。
- 触发原因：核心路径、低置信、近期变更、风险、覆盖不足或未解问题。
- 证据入口：代码、测试、配置或文档。
- 下一步：下一次打开该研究任务时要做的具体动作。
```
