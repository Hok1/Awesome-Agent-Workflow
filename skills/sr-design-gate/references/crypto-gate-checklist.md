# SR Design Gate 密码学专项检查清单

仅当 SKILL 的密码学适用性判定结论为 `涉及` 时执行本清单。冲突分级、证据有效性
和检查规则沿用 `gate-checklist.md`，本清单不重复定义。

## 算法类别与敏感参数口径

适用性判定和识别完备性检查以下列口径为对照基准：

- 算法类别：对称加密、非对称加密、哈希、消息认证码（MAC）、密钥派生（KDF）、
  随机数生成（DRBG），以及基于它们的密钥协商、签名和后量子算法。
- 敏感参数类型：对称密钥、私钥、口令、共享秘密、签名一次性随机数（k 值）、
  KDF 输入材料（IKM/salt）、DRBG 内部状态、会话密钥，以及中间计算敏感数据
  （模幂中间值、派生中间状态）。
- 安全关键元数据：密钥引用、安全属性（`sensitive`/`extractable` 等）、
  生命周期状态、审计元数据和受保护导出材料。

## 1. 适用性判定

判定输入是 **SR 设计正文**——SR 涉及密码算法但未做敏感参数分析，正是本专项
要拦截的 `不通过` 场景，不得因正文没写就判 `不适用`。

- 对照上节算法类别，在 SR 设计的接口、数据、配置、依赖和 DFX 安全章节中
  检索，形成命中/未命中结论。
- 判定为 `不涉及` 时必须记录检索依据（检索了哪些章节、对照了哪些类别）；
  无依据的 `不涉及` 记 P1，且敏感参数安全维度不得判 `不适用`。
- SR 设计自身声明"不涉及密码算法"但未附检索或排查说明：同样记 P1。

## 2. 识别完备性

检查 SR 设计是否完成敏感参数识别，缺项即维度未达标：

- 涉及的密码算法全部列出，含对称、非对称、哈希、MAC、KDF、DRBG。
- 每种算法涉及的敏感参数类型已对照上文口径逐一识别。
- 每个敏感参数的来源（用户输入、系统生成、外部获取、协商产出）和存储位置
  （内存、堆、磁盘、配置、密钥库）已识别。
- 中间计算敏感数据（签名 k 值、模幂中间值、派生中间状态）已识别。
- 密钥引用、安全属性、生命周期状态、审计元数据等安全关键元数据已识别。

## 3. 架构级密钥管理决策

以下属于 SR 层的模块职责与契约内容，缺失按冲突分级口径记 P0/P1：

- 每个密钥定义了所有者、用途、允许操作、访问角色和最小权限边界。
- 密钥生命周期状态及合法转换（生成、启用、轮换、吊销、泄露、销毁等）已设计。
- 同一密钥不得跨加密、签名、认证、密钥包装等不兼容用途复用（违反记 P0）。
- 密钥泄露、疑似泄露、吊销时的联动处置（受影响数据、证书、上层会话）有方案。
- 使用 KMS、HSM、Provider 或远程密钥服务时，信任边界、认证、会话管理、
  故障切换和不可用时的失败策略已明确。
- 私钥存储方案为加密存储或专用密钥库，无明文持久化（违反记 P0）。

## 4. 跨模块传输契约

- 敏感参数出现在模块交互接口上时，接口契约必须声明保护方式：加密保护、
  安全通道或禁止明文跨进程传递；未声明记 P1，明确允许明文传递记 P0。
- 结合关键事实台账中标记的涉敏接口逐一核对，不得抽样。

## 5. 导出接口策略

SR 设计存在私钥导出、序列化、复制、备份或恢复类接口时：

- 必须以"默认不可导出"为基线，明确允许导出、仅允许加密导出或禁止导出的分类。
- 允许导出时必须定义授权角色、用途限制和加密/完整性保护要求。
- 存在此类接口但完全未声明导出策略：记 P0。
- 不存在此类接口时本节 `不适用`，须写明检索依据。

## 6. 下游传导检查

内存清零、随机源安全、算法特殊要求（nonce/IV/k 值）、fork 与进程复制安全、
导出接口实现细节等属于模块详设与开发阶段的检查项，本专项**不复核其实质内容**，
只确认传导不断链：

- SR 设计已把适用的实现级安全要求标注为下游输入：体现在 AR 范围、
  模块设计约束或验收标准中，确保 module 设计与 task-dev 阶段能够承接。
- 已判定涉及密码算法但 AR 拆分和验收标准中完全没有安全要求的传导落点：记 P1。
