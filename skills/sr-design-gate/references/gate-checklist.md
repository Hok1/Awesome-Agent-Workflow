# SR Design Gate 检查清单

## 准入维度

逐项判定为 `达标`、`未达标` 或 `不适用`：

| 维度      | 必查内容                                                              |
| ------- | ----------------------------------------------------------------- |
| 输入有效性   | 架构基线、SR 设计、模板结构和历史门禁结果可读                                          |
| 需求完整性   | 目标、范围、主流程、异常和兼容场景完整；逐项写明变更前与变更后，不得只描述新方案                          |
| 范围清晰性   | 范围内、范围外、依赖方职责及不承担内容明确                                             |
| 架构一致性   | 模块职责、所有权和依赖方向符合软件架构                                               |
| 契约完整性   | 接口、数据、状态、配置、错误、超时、重试、幂等和兼容策略完整；边界收紧或放宽已判定兼容性并明确通知对象               |
| AR 可拆分性 | 每个 AR 有稳定 ID、标题、边界、价值、模块、依赖和交互契约                                  |
| 需求追溯性   | 每项需求有且仅有明确 AR 承接，无遗漏和冲突性重复                                        |
| 验收闭环性   | 功能、异常、接口、数据、DFX 和关键风险均有可执行验收                                      |
| 决策与风险闭环 | 待确认项、假设以及安全、迁移、回滚和降级风险闭环；升级、降级和混合版本互操作已分析                         |
| 变更影响分析  | 按「变更影响分析检查」和「分级一致性」规则检查 SR 设计的变更影响分析与 E0-E3 分级；全新系统无既有行为基线时可判不适用并说明依据  |
| 敏感参数安全  | 按 SKILL 的密码学适用性判定执行 `crypto-gate-checklist.md`；判定不涉及时记录检索依据后可判不适用 |
| 文档内部一致性 | 同一事实在文字、表格、图、AR 和验收章节中一致                                          |

## 变更影响分析检查

检查 SR 设计正文（含其变更影响说明，无论以章节还是附表形式呈现）是否完成变更
影响分析。SR 设计侧没有独立的自检交付物；缺失的分析视为设计缺口，整改回
sr-design 补齐：

### 分级口径

- E0：外部可观察行为、结果和资源约束均不变。
- E1：存在兼容性对外变更，既有用法仍受支持。
- E2：仅部分条件下兼容或行为发生变化。
- E3：既有合法用法不再兼容。
- 设计阶段证据不足时标记 `待确认`，不得直接形成确定分级。

### 覆盖面检查

SR 设计涉及既有行为变更时，正文必须覆盖以下方面，缺失即维度未达标：

- 分析基线：所依据的版本、配置、平台和既有行为基线；逐项写明变更前与变更后。
- 对外变化识别：API/接口签名、参数与返回语义、错误码、输出格式与顺序、
  副作用与状态、调用顺序与幂等、并发与线程安全、CLI/配置/环境变量默认值、
  协议与持久化格式、日志与可观测输出；逐项判断既有正确用法是否仍得到
  语义等价结果，并明确受影响对象和所需适配。
- 上层影响传导：从变更点追踪直接调用方、封装/适配/框架层直至上层功能和
  用户场景；检查资源与性能变化向超时、重试、限流和容量的传导。
- 边界与异常：边界收紧或放宽的兼容性判定；失败语义、部分执行副作用、
  失败后能否重试及状态恢复。
- 资源与性能：先分级为 `较小`/`较大`/`待确认`——`较小` 允许有依据的定性或
  数量级评估；`较大` 或 `待确认` 必须给出变更前后预估、增量或比例、峰值和
  估算模型，并说明变化原因；无论大小均须定义开发阶段的实测计划。
- 兼容与发布：源码/二进制/行为/数据/配置兼容性分别判断；升级、降级、回退和
  混合版本互操作；调用方适配、文档和发布说明更新；灰度观测与回退触发条件。

### 证据规则

- 每项 `无影响`/`不变` 结论必须有可反查的证据（设计章节、现有接口或代码位置、
  调用链检索结果、计划测试用例、估算依据或评审结论）；无证据记 P1。
- 无效证据否决：`改动行数小`、`仅内部重构`、`接口签名未变`、`单元测试通过`
  均不得单独作为无对外影响的证据；以此类证据支撑的结论视同无证据。
- 证据时限：设计阶段证据不得要求代码提交、PR、运行日志或性能实测结果；
  Gate 也不得以缺少实测数据为由判定未达标。
- `无上层影响` 结论必须写明追踪边界、调用方清单和证据；缺失记 P1。
- `待确认` 项必须有责任人和完成时间，并计入待确认问题。

## 分级一致性

按上节分级口径复核以下三条：

1. 分级结论与逐项分析矛盾（如存在已确认的非兼容对外变更却定级 E0/E1）：记 P0。
2. 存在 E3 非兼容变更但未给出审批计划、迁移或回退设计：结论不得为 `通过`。
3. 对外行为、上层影响、资源或性能任一项为 `待确认` 却形成确定的 E0 结论：
   结论不得为 `通过`。

## 关键事实台账

在本轮分析上下文中全量抽取并临时记录。该台账不是独立交付物；仅在触发门禁报告生成规则时，把相关事实、问题和证据写入报告：

- 模块：名称、职责、不承担范围、上下游和依赖方向。
- 接口：调用方、提供方、协议、同步方式、请求响应、错误、超时、重试、幂等和兼容；
  涉及密码算法时同步标记传递敏感参数的接口。
- 数据：对象、字段、类型、必填、默认值、单位、所有者、生命周期和一致性要求。
- 状态：状态集合、初始状态、触发条件、转换方向、失败和终止状态。
- 配置：名称、类型、默认值、单位、生效范围和安全约束。
- AR：ID、标题、范围、范围外、模块、前置依赖和交互接口。
- 指标：SLA、超时、性能、并发、可用性、重试次数和降级条件。

## 权威定义和交叉位置

先从当前 SR 设计模板识别承担下列语义职责的章节，再按语义进行交叉核对，不依赖固定章节号。当前模板缺少适用的语义位置时，应记录模板完整性问题。

| 信息类型 | 权威语义位置 | 交叉检查语义位置 |
|---|---|---|
| 需求范围和目标 | 描述需求背景、目标以及范围内/外的章节 | 功能设计、需求追溯和整体验收 |
| 模块边界 | `software_architecture.md` 以及描述整体架构和模块职责的章节 | 现状模块、模块交互、AR 概览、AR 边界和架构变更 |
| 对外接口 | 描述对外接口契约的章节 | 主成功场景、模块交互、异常、AR 接口和验收 |
| 数据模型 | 描述数据模型、数据流转和存储的章节 | 现状数据、数据流、状态、AR 接口和验收 |
| 状态转换 | 描述状态机的章节 | 主场景、流程、关键步骤、异常和验收 |
| 异常语义 | 描述异常、冲突和兼容场景的章节 | 对外接口、主场景、AR 接口和验收 |
| DFX 指标 | 描述可靠性、安全性、可服务性和性能的章节 | 外部依赖、配置、风险和验收 |
| AR 身份和范围 | 描述 AR 概览和边界的章节 | AR 依赖、AR 接口和需求追溯 |
| AR 依赖 | 描述 AR 依赖关系的章节 | 模块交互、AR 接口和 AR 边界 |
| AR 交互契约 | 描述 AR 间交互的章节 | 对外接口、模块交互、数据和验收 |
| 配置项 | 描述配置设计的章节 | 外部依赖、异常、DFX 和验收 |
| 验收标准 | 描述验收总览和黑盒用例的章节 | 反向覆盖需求、接口、功能、DFX 和 AR |

## 一致性链

逐对象交叉核对以下链路：

1. 主成功场景 ↔ 模块交互时序 ↔ 流程图 ↔ 关键步骤 ↔ 数据流 ↔ 状态机 ↔ 黑盒验收。
2. 对外接口 ↔ 模块交互 ↔ AR 间接口 ↔ 数据设计 ↔ 异常场景 ↔ 验收断言。
3. AR 概览 ↔ 依赖图 ↔ AR 间接口 ↔ AR 边界 ↔ 需求追溯。
4. 外部依赖 SLA ↔ DFX ↔ 配置 ↔ 风险处理 ↔ SR 验收标准。
5. 变更影响分析 ↔ SR 设计正文 ↔ E0-E3 分级结论 ↔ 风险与验收章节：
   分析中承认的每项对外变化，正文契约、兼容策略和验收章节必须有对应落点；
   分析与正文冲突按冲突分级口径处理。

AR ID、标题、边界、依赖、接口、字段、类型、状态、错误码、配置、单位、阈值、主流程和关键失败路径必须全量检查。

## 冲突分级

- P0：会造成实现分叉、接口不兼容、数据错误或错误验收，例如字段类型、状态转换、依赖方向、同步模型、事务或安全策略冲突；密码学场景中密钥跨用途复用、敏感参数明文持久化、密码学用途使用非 CSPRNG 随机源亦属 P0。
- P1：使开发或测试无法确定真实设计，例如默认值、超时、重试、范围、流程步骤、DFX 或验收阈值冲突。
- P2：不影响实现的明确表达差异，可作为非阻断改进项。

存在 P0/P1 时结论必须为 `不通过`。缺少权威决策且无法判断时结论为 `阻塞`。
