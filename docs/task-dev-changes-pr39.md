# 软件变更影响分析开发阶段自检 Checklist

## 1. 使用说明

本 Checklist 供 `task-dev` 在代码实现、测试验证和任务交付阶段使用，承接 `sr-design` 的[《软件变更影响分析设计阶段自检 Checklist》](<skill-dir>/sr-design/reference/change-impact-analysis-checklist.md)。它以任务文件和上游设计中的变更影响分析为基线，通过实际代码差异和测试证据验证设计结论。`sr-design` 不加载或执行本文件。

结果栏填写：

- 通过：实际实现和验证满足检查要求，且证据充分；
- 不通过：存在缺失或不符合项；
- N/A：不适用，必须说明原因；
- 待确认：证据不足，必须明确阻塞影响和后续处理方式。

证据位置可填写任务文件章节、代码文件和行号、完整差异、调用链搜索结果、提交或 PR、自动化测试、日志、性能原始数据或评审结论。

执行规则：

- 每个任务均逐项判断适用性；不得只检查主要功能文件。
- 任一检查项不得只填"通过"而不提供证据。
- 实现偏离设计时，先停止并按 `task-dev` 的冲突处理规则向用户报告，不得用开发阶段结论静默改写设计。
- 存在未关闭的 P0、P1 或影响当前任务验收的"待确认"项时，不得将任务标记为 `Completed`。
- "接口签名未变"、"单元测试通过"或"代码改动很小"均不能单独证明无对外影响。
- 只有实际实现拟维持 E0 结论时才执行第 4 章快速反证；任一答案为"能"或"待确认"时，必须重新判定影响级别并回补相关设计或任务记录。

# 基础密码算法敏感参数开发阶段自检 Checklist

## 1. 使用说明

本 Checklist 供 `task-dev` 在代码实现和测试验证阶段使用，承接 `sr-design` 的[《基础密码算法敏感参数与特殊要求分类清单》](<skill-dir>/sr-design/reference/sensitive-param-classification.md)和[《基础密码算法敏感参数设计自检 Checklist》](<skill-dir>/sr-design/reference/sensitive-param-checklist.md)。若任务文件、上游设计或实际代码涉及基础密码算法、密钥/私钥、口令、令牌、随机数、KMS/HSM、私钥导入导出或 Fork/进程复制场景，必须加载本文件；否则可整体标记为 N/A，并记录代码和设计检索依据。

以当前任务文件和上游 DFX 6.5 为设计基线。优先沿用上游 6.5.1/6.5.2 的 `SP-*` 参数编号，将每个参数映射到实现位置和验证证据；若上游未编号，应先建立等价的任务内参数编号，不得只按算法或模块给出笼统结论。若实现中发现设计未覆盖的私钥导出、密钥管理或 Fork 风险，暂停开发并按 `task-dev` 的设计冲突规则报告，不得通过代码自行决定安全策略。

结果标记：

- 通过：实际实现和验证符合设计，且有代码、差异或测试证据；
- 不通过：实现或验证不符合设计；
- N/A：不适用，必须说明检索范围和原因；
- 待确认：设计或证据不足，必须说明阻塞影响。

门禁规则：

- 任一适用项不得只填“通过”而没有证据。
- 私钥越权或意外导出、明文私钥泄露、父子进程随机/nonce 状态复用、密钥状态绕过或敏感数据未清零属于阻断问题。
- 存在未关闭的阻断问题或影响验收的待确认项时，不得将任务标记为 `Completed`。
