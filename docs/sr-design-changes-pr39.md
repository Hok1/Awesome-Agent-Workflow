# 软件变更影响分析设计阶段自检 Checklist

## 1. 使用说明

本 Checklist 与软件变更影响分析配套使用，仅供 `sr-design` 在设计阶段识别和约束变更影响。

结果栏填写：

- 通过：已经完成设计分析，且证据满足要求；
- 不通过：存在缺失或不符合项；
- N/A：不适用，必须在证据或问题栏说明原因；
- 待确认：证据不足，必须指定责任人和完成时间。

证据位置可填写设计章节、现有接口或代码位置、调用链搜索结果、计划测试用例、估算依据或评审结论。不得要求尚未进入开发阶段才能产生的提交、PR、运行日志或性能实测结果。

门禁规则：

- 任一检查项不得只填"通过"而不提供证据。
- 存在 E3 非兼容变更但未给出审批计划、迁移或回退设计时，不得通过设计自检。
- 对外行为、上层影响、内存或性能标记"待确认"时，不得形成确定的 E0 设计结论。
- "接口签名未变""单元测试通过"或"代码改动很小"均不能单独证明无对外影响。

| 项目 | 填写内容 |
| --- | --- |
| 变更名称 |  |
| 需求、缺陷或任务编号 |  |
| 模块和目标版本 |  |

| 自检阶段 | 设计阶段 |
| 自检人员 |  |
| 自检日期 |  |
| 设计阶段初步分级 | E0 / E1 / E2 / E3 / 待确认 |

分级口径：E0 表示外部可观察行为、结果和资源约束均不变；E1 表示存在兼容性对外变更但既有用法仍受支持；E2 表示仅部分条件下兼容或行为发生变化；E3 表示既有合法用法不再兼容。设计阶段证据不足时标记为"待确认"，由开发阶段结合实际实现复核。

# 基础密码算法敏感参数设计自检 Checklist

本自检清单参考 OpenHiTLS 编码安全规范和基础密码算法库设计实践，用于涉及基础密码算法（对称加密、非对称加密、哈希、消息认证码、密钥派生、随机数生成等）实现或使用的功能设计、模块详细设计和设计门禁阶段。当确认本次需求涉及密码算法时，按本清单逐项自检，发现不满足项应标记为设计缺口并补充设计或回补上游。实际代码、运行行为和测试执行由 `task-dev` 的[《基础密码算法敏感参数开发阶段自检 Checklist》](<skill-dir>/task-dev/reference/sensitive-param-checklist.md)负责，不在本清单中验证。

本清单是辅助自检工具，不替代各阶段 SKILL 中定义的工作流和质量标准。使用前应先加载[《基础密码算法敏感参数与特殊要求分类清单》](./sensitive-param-classification.md)，识别本次需求涉及的敏感参数类型和特殊要求。

使用方式：

1. 先判断本次需求是否涉及基础密码算法的实现或使用。如不涉及，本清单不适用，在设计文档中注明"不涉及密码算法，本清单不适用"。
2. 如涉及，加载[《基础密码算法敏感参数与特殊要求分类清单》](./sensitive-param-classification.md)，识别涉及的算法类别和对应的敏感参数类型。
3. 按本清单逐项自检，对每项标记：✓ 已满足 / ✗ 不满足需补充 / ○ 不适用（说明原因）。
4. 不满足项应转化为设计决策点或缺口记录，纳入对应阶段的问题澄清、TOBE 设计补充或门禁整改清单。

## 1. 敏感参数识别

- [ ] **P1-01**：是否已列出本次需求涉及的全部密码算法（含对称、非对称、哈希、MAC、KDF、DRBG）？
- [ ] **P1-02**：是否已对照[《基础密码算法敏感参数与特殊要求分类清单》](./sensitive-param-classification.md)的分类表，逐一识别每种算法涉及的敏感参数类型？
- [ ] **P1-03**：是否已识别每个敏感参数的来源（用户输入、系统生成、外部获取、协议协商产出）？
- [ ] **P1-04**：是否已识别每个敏感参数的存储位置（内存变量、堆分配、磁盘文件、配置项、密钥库）？
- [ ] **P1-05**：是否已识别中间计算过程中的临时敏感数据（签名 k 值、模幂中间值、派生过程中间状态）？
- [ ] **P1-06**：是否已识别密钥引用、安全属性、生命周期状态、审计元数据和受保护导出材料等安全关键元数据？
- [ ] **P1-07**：是否已识别 Fork 或等价进程复制时可能被复制的 DRBG、nonce/计数器、算法上下文和外部设备会话状态？

## 2. 敏感参数生命周期与密钥管理

- [ ] **P2-01**：6.5.1 中每个敏感参数是否具有唯一参数编号，并在 6.5.2 中按同一编号逐项分析且无遗漏？是否按参数类型定义适用的生命周期或存续路径，并对不适用阶段说明原因？
- [ ] **P2-02**：是否为每个密钥定义所有者、用途、允许操作、访问角色和最小权限边界？
- [ ] **P2-03**：是否定义密钥状态及合法转换（生成、待激活、启用、暂停、轮换、吊销、泄露、归档、销毁）？
- [ ] **P2-04**：是否定义密钥使用周期、轮换条件、版本切换、旧密钥保留期限和并发使用窗口？
- [ ] **P2-05**：是否禁止同一密钥跨加密、签名、认证、密钥包装等不兼容用途复用？
- [ ] **P2-06**：外部导入密钥是否定义来源认证、完整性校验、算法与长度校验以及安全属性继承规则？
- [ ] **P2-07**：是否按密钥类型明确备份、恢复、托管和归档策略，并明确临时私钥、签名一次性秘密等不得备份或归档的材料？
- [ ] **P2-08**：是否定义密钥泄露、疑似泄露、吊销、替换以及受影响数据、证书和上层会话的联动处置？
- [ ] **P2-09**：使用 KMS、HSM、Provider 或远程密钥服务时，是否定义信任边界、认证、会话管理、故障切换和不可用时的失败策略？
- [ ] **P2-10**：是否设计密钥标识、版本、状态、使用者和关键操作的库存与审计记录，且审计信息不包含密钥材料？
- [ ] **P2-11**：私钥的存储方案是否为加密存储或专用密钥库（禁止明文持久化存储）？
- [ ] **P2-12**：私钥文件的访问权限是否已设计为仅属主可读（如 `0600`）？
- [ ] **P2-13**：密钥协商共享秘密是否设计为仅用于 KDF 派生，派生完毕后立即清零（禁止直接用作加密密钥）？
- [ ] **P2-14**：会话密钥是否设计为会话结束后立即清零，禁止跨会话复用？
- [ ] **P2-15**：口令类参数是否设计为接收后仅用于校验/派生，校验/派生完毕后立即清零，禁止明文持久化？

## 3. 内存清零

- [ ] **P3-01**：每个敏感参数使用完毕后是否设计了内存清零操作？
- [ ] **P3-02**：清零方法是否使用了不可被编译器优化跳过的方式（volatile memset / memset_s / 项目安全清零函数），而非普通 `memset` + `free`？
- [ ] **P3-03**：清零后是否设计了指针置 NULL？
- [ ] **P3-04**：中间计算缓冲区（签名 k、模幂中间值等）是否设计为计算完成后立即清零？
- [ ] **P3-05**：函数返回前，是否检查了所有局部敏感变量是否已清零？
- [ ] **P3-06**：错误路径/异常退出路径中，是否同样设计了敏感参数的清零（不能只在正常路径清零）？

## 4. 日志与异常屏蔽

- [ ] **P4-01**：日志输出是否确认不包含任何敏感参数原始值（密钥字节、口令明文、签名 k 值、共享秘密等）？

- [ ] **P4-03**：异常信息和错误堆栈是否确认不泄露敏感参数值？
- [ ] **P4-04**：调试/诊断输出是否有独立开关控制，默认关闭，禁止在生产环境开启后输出敏感内容？

## 5. 随机源安全

- [ ] **P5-01**：密钥生成、nonce/IV 生成、签名 k 值生成是否设计为使用密码学安全随机源（CSPRNG/DRBG）？
- [ ] **P5-02**：是否禁止使用 `rand()`、时间戳、PID 等非密码学安全随机源作为密码学用途的输入？
- [ ] **P5-03**：DRBG 是否设计了状态管理和重播种策略（遵循 NIST SP 800-90A 或 GM/T 0102）？
- [ ] **P5-04**：AES-GCM nonce 是否设计了在同一密钥下不重复的保障机制？
- [ ] **P5-05**：AES-CBC IV 是否设计为不可预测（非固定值）？
- [ ] **P5-06**：ECDSA/SM2 签名 k 值是否设计为每次签名独立随机生成，使用后立即清零？

## 6. 传输安全

- [ ] **P6-02**：私钥跨进程传递是否设计为禁止明文传递？
- [ ] **P6-03**：密钥分发流程是否设计了端到端加密或密钥协商机制？
- [ ] **P6-04**：是否禁止在共享目录中创建包含敏感数据的临时文件？

## 8. 私钥导入导出接口安全

当功能不涉及私钥导入、导出、序列化、复制、备份或恢复接口时，本节可整体标记为 N/A 并说明检索依据。

- [ ] **P8-01**：是否识别所有可能使私钥离开原安全边界的接口，包括显式导出、通用序列化、复制、备份、恢复和调试接口？
- [ ] **P8-02**：是否按密钥类型明确允许导出、仅允许加密导出或禁止导出，并以默认不可导出作为无需导出场景的默认策略？
- [ ] **P8-03**：导出操作是否定义调用者身份、授权角色、用途限制、最小权限和必要的二次确认或审批机制？
- [ ] **P8-04**：是否禁止无明确业务必要性的明文私钥导出；确需导出时是否定义可信通道、密钥包装或等价保护以及风险接受条件？
- [ ] **P8-05**：复制、派生、导入、恢复或格式转换是否保证 `sensitive`、`extractable` 等安全属性不能被静默降级？
- [ ] **P8-06**：是否定义导出格式、加密与完整性保护算法、包装密钥用途和强度、版本兼容与反回滚要求？
- [ ] **P8-07**：是否认证导出目标或接收方，并防止导出数据被替换、重放或发送到错误主体？
- [ ] **P8-08**：导出缓冲区是否定义所有权、长度、释放方、有效期和不可优化清零要求，禁止产生无主副本？
- [ ] **P8-09**：长度探测、分段导出、缓冲区不足、权限失败和其他错误路径是否保证不返回部分私钥或泄露私钥属性？
- [ ] **P8-10**：是否设计导出、拒绝导出和策略变更的审计记录，且日志、错误和审计中不包含私钥材料或可还原片段？
- [ ] **P8-11**：是否限制导出频率、并发和重复调用，避免接口被批量滥用或在失败重试中产生额外副本？
- [ ] **P8-12**：私钥导入是否验证加密与完整性保护、算法参数、密钥长度、公私钥匹配关系和目标安全属性？

## 9. Fork 与进程复制安全

当目标平台或宿主进程不支持 `fork`/`vfork`、且不存在等价的进程快照或预派生工作进程模型时，本节可整体标记为 N/A 并说明依据。

- [ ] **P9-01**：是否明确运行环境可能发生 `fork`、`vfork`、预派生 worker 或等价进程复制的入口和时机？
- [ ] **P9-02**：是否明确仅支持 `fork` 后立即 `exec`，还是允许子进程继续执行密码操作；不支持的模式是否有可检测的失败策略？
- [ ] **P9-03**：是否识别子进程会继承的私钥、会话密钥、DRBG、nonce/计数器、算法上下文、锁、线程局部状态、文件描述符、共享映射和设备会话？

- [ ] **P9-07**：算法上下文、Provider/Engine、KMS/HSM 连接和设备会话是否在子进程中失效并重新初始化，禁止直接复用父进程句柄？
- [ ] **P9-08**：是否处理多线程进程 fork 后遗留锁、回调和线程局部状态，避免子进程死锁、未定义行为或绕过初始化？
